## はじめに

各種ベンチマークを見る限り、Opus 4.8に軍配が上がる結果が多いようです。ところが実際にSNSを覗いてみると、少し違う声も聞こえてきます。特に日本語で指示を出したときの挙動が「なんとなく怪しい」という理由で、Opus 4.7やOpus 4.6をメインで使い続けている、という人も一定数いるようです。

正直なところ、自分で使っていても「このモデルだけ劇的に違う」と感じる場面はそこまで多くなく、そのときの気分で違うモデルに切り替えながら作業をしていました。

そんな折、Fable 5用に課金していたMaxプラン(×20)の週次利用枠が余りそうになりました(Fable 5自体の利用上限は、当然のようにとっくにお亡くなりになっていました)。せっかく枠が余っているならと思い、同じプロンプトをOpus 4.6 / 4.7 / 4.8にそれぞれ投げて、その出力をCodexくんに分析してもらうことにした、というのが今回の検証のきっかけです。汎用ベンチマークのスコア比較ではなく、実際に運用しているShopifyアプリテンプレートのリポジトリを題材にした、**repoに閉じた実務評価**にしてみました。

目的は「どのモデルが優秀か」を決めることではありません。「どのタスクをどのモデルに投げると事故りにくいか」を、手元の材料で確かめてみたかったんです。先に結論を言ってしまうと、3モデルには結構はっきりした役割分担があって、1モデルに全部任せるより使い分けたほうが良さそう、という話になりました。

ちなみにこれは特定の1リポジトリ・特定のプロンプト群での観察であって、モデルの一般的な優劣を主張するものではありません。その点は最後の「注意点」でも改めて触れます。

## 評価対象のrepo

比較に使ったのは、Shopifyアプリテンプレートのリポジトリです。特徴は次の通りです。

- `AGENTS.md` に強い運用ルールがあり、harness tickets を先に完了させてからドメイン機能を実装する構成になっている
- 「1セッション1ticket」が原則で、非自明な変更の前には `plans/<ticket-id>.md` を作ることが求められる
- auth / billing / webhooks / retention / state machine / route contract / review metadata / infra selection に触る変更には ADR が必須
- `shopify.app.toml`、scope、billing truth、webhook policy、privacy/delete contract は ticket に明示がない限り変更禁止
- billing truth は Partner API の `activeSubscription`、scope truth は `currentAppInstallation.accessScopes` であり、webhook payload や redirect の query param は truth として扱わない
- Orders / Customers / Discounts は launch scope 外
- 標準チェックは `pnpm check`、実装変更には test / contract fixture / smoke のいずれかの変更を伴わせる

このrepoを選んだのは、単純なコード生成力より「repoのgovernance・ADR・truth sourceをちゃんと理解できるか」が問われる作りになっているからです。Shopify固有の判断は公式docsを正本にする必要があって、思い込みで答えると普通に間違えます。LLMにとっては「知ってるフリができているだけか、実際に読み取れているか」を試す、ちょうどいい試金石になっています。

## 評価指標

以下の13指標で、0〜4の5段階評価をしました。

| カテゴリ | 指標 |
|---|---|
| 理解・遵守 | 要求理解 / 制約遵守 |
| 正確性 | 正確性 / 根拠性 / 完全性 |
| 実務適性 | 実用性 / 判断力 / 効率 |
| メタ認知 | 不確実性管理 / 検証姿勢 / 一貫性 / 自律性 |
| コミュニケーション | コミュニケーション品質 |

- **0** = 不合格
- **1** = 大きな問題あり
- **2** = 普通
- **3** = 良い
- **4** = 非常に良い

採点そのものは、各モデルの出力をCodex(CLI)に読み込ませて行いました。自分ひとりの主観だけで採点すると、無意識に贔屓が入りそうだったので、第三者的な立場のツールに判定を任せています。

この記事では「賢さ」そのものより「失敗しにくさ」を軸に見ています。理由は単純で、実務で困るのは「難しい質問に答えられない」ことより、「もっともらしい顔でrepoのルールを踏み越える」ことのほうが圧倒的に多いからです。制約遵守・根拠性・不確実性管理・検証姿勢あたりを重めに見ているのもそのためで、要は「地雷を踏まない能力」を測る項目だと思っていただければと思います。

## 検証したプロンプト

以下の7種類のプロンプトで比較しました。いずれも実際にこのrepoで発生しうる依頼をベースにしています。

1. **リファクタリング案レビュー** — コードの目利きと、repoルールへの接続を見ます
2. **オンボーディング改善提案のレビュー** — UX判断とtruth sourceとの整合性を見ます
3. **billing / entitlement 障害調査** — 一次情報の切り分けと不確実性の扱いを見ます
4. **PRレビューコメントへの対応方針分類**(accept / reject / needs-confirmation) — 迎合しない判断力を見ます
5. **Shopify App Store 提出前の Go / No-Go 判断** — リスク評価と抜け漏れ検出力を見ます
6. **Setup Health 画面の要件定義** — 曖昧な要求から要件を切り出す力を見ます
7. **fork先機能をテンプレ本体へ昇格するための大規模設計相談** — アーキテクチャ判断とADRへの接続力を見ます

軽いレビュー系から、governanceの理解がガチで問われる設計相談まで、だんだん負荷を上げていったのがポイントです。

## モデル別の観察

### Opus 4.6 — 実装者型

小粒なリファクタ候補出しが得意な、いわば「手を動かすタイプ」です。重複関数やDRY違反、局所的な改善点を具体的に指摘してくる力は高く、回答も読みやすく実用的でした。

ただ、repoのADR・ticket・governance・truth sourceへの接続はやや浅めでした。コードそのものの品質にはよく目が行くのですが、「なぜこのルールがあるのか」まで踏み込んで判断する場面では、少し弱さが出ていました。

具体例です。

- 最初のリファクタリングレビューでは、`resolveErrorCode` の重複、funnel処理の重複、webhook分岐の整理など、具体的な改善候補をきちんと出してくれました
- ただし「1セッション1ticket」やADR必須領域、scope変更禁止といったrepo固有ルールへの言及は弱く、レビューがコード単体の話に閉じがちでした
- `SHOPIFY_APP_HANDLE` 未設定時に `/app/pricing` がfail-fastする設計を、深く疑わずにacceptしてしまう場面があり、repoの設計意図と少しズレた判断をしていました
- PRレビューコメントへの対応方針分類では、rejectすべき提案を受け入れ気味に倒す傾向も見られました

向いている用途は、小粒なリファクタ候補出し・コードスメルの発見・実装前の改善案リストアップです。逆に、ADRやgovernanceが絡む設計判断、Shopify審査やbilling / webhook / scopeの安全性判断、大規模な方針策定には向いていません。

### Opus 4.7 — テックリード型

今回の検証でいちばんの本命でした。repoの制約・ADR・ticket・docs・testsを安定して扱え、accept / reject / needs-confirmation の切り分けが特に強かったです。レビューコメントに対する迎合耐性も高く、「言われたから受け入れる」ではなく、repoのルールに照らして判断する姿勢が一貫していたのが印象的でした。

具体例です。

- PRレビューコメントの対応方針分類では、フルacceptはゼロ件でした。7件中、docs / smoke / validation に関わる1件だけをpartial acceptとし、他はrejectまたはneeds-confirmationに切り分けていました
- Setup Health画面の要件定義では、merchant向けUIとfork operator向けreadinessという、性質の異なる2つの利用者像をきちんと分離して整理できていました
- billing障害調査では、plan handle allowlist、Partner APIの環境変数、force refresh routeの穴、shopGid未補完といった複数の疑いをバランスよく分解し、一つに決め打ちしませんでした
- Go / No-Go判断では、release gate、billing E2E、webhook、reviewer packet、listing assets まで漏れなく網羅していました

要件をticket / ADR / docs / testsという形に落とし込むのがうまく、governanceの強いrepoでの通常運用にはかなり向いています。強いて弱点を挙げるなら、やや重めで、軽い雑談レベルの相談でもticket化・ADR化したがる癖があることです。「ちょっと聞きたかっただけなんだけどな」という場面では、ややオーバースペックに感じることもあります。

向いている用途は、普段のrepo作業相談・PRレビュー対応・障害調査・要件定義・ticket/ADR/docs/testsへの落とし込みなど、governanceが強いrepoでの日常業務全般です。

### Opus 4.8 — アーキテクト / 戦略レビュー型

大規模設計、前提矛盾の検出、リリース判断で一番の強みを発揮しました。刺さったときの洞察の鋭さは、3モデルの中でも頭一つ抜けている感じがします。repoの思想や過去の設計判断を踏まえて、本質的な境界線を見つけ出す力がありました。ただその分、脱線・過剰な抽象化・断定の強さといったリスクもきちんとあります。

具体例です。

- Go / No-Go判断において、「そもそもこのworking directoryは、未forkのcanonical templateなのでは?」という前提矛盾を見抜きました。これは他の2モデルでは出てこなかった指摘です
- fork先機能をテンプレ本体へ昇格する大規模設計相談では、過去の昇格成功例だけでなく、あるドメインがarchiveされた「demotion(降格)」の前例まで引き合いに出し、判断材料に厚みを持たせていました
- external SaaSを標準として組み込まない理由を、既存のtelemetryがstdout EMFで外部依存ゼロで完結している事実から筋道立てて説明していました
- 一方で、最初のリファクタリングレビューでは、求められていたリファクタ案そのものより、サブエージェント分割の話に脱線する場面もありました
- 障害調査では、「fallback logが無ければhandle不一致のはず」のように、確認前提の事実を断定気味に語る場面があり、不確実性管理の観点ではやや扱いに注意が必要でした

向いている用途は、大規模設計相談、Go / No-Go判断、4.7が出した設計案へのセカンドレビュー、前提矛盾のチェック、repoの思想整理です。逆に、軽いリファクタ相談や「短く直接答えてほしい」タスク、厳密な不確実性管理が求められる障害調査の一次回答役には向いていません。

## 用途別の使い分け

総合順位としては 4.7 > 4.8 > 4.6 という結果になりましたが、これは「4.7が万能」という意味ではなく、governanceの強いrepoでの総合力がたまたま高かった、というだけの話です。実際には用途ごとに向き不向きがはっきり分かれています。

| 用途 | 推奨モデル |
|---|---|
| 普段のrepo作業相談 | Opus 4.7 |
| PRレビューコメント対応 | Opus 4.7 |
| 障害調査 | Opus 4.7、補助でOpus 4.8 |
| Go / No-Go判断 | Opus 4.8、確認にOpus 4.7 |
| 大規模設計・方針策定 | Opus 4.8、実務化にOpus 4.7 |
| 小粒リファクタ候補出し | Opus 4.6 |
| 曖昧な新機能の要件定義 | Opus 4.7 |

大づかみに言うと、4.6は「手」、4.7は「日々の判断」、4.8は「大局観と前提の疑い直し」を担当させると一番ハマる、というのが実際に使ってみた感覚です。

## 気づき

今回の検証を通じて得られた気づきを整理します。

**モデルごとの「賢さ」より「失敗パターン」を把握しておくほうが、実務ではずっと役に立ちます。** どのモデルも単発の質問には十分なレベルで答えてくれます。差が出るのは、repoのルールと矛盾する提案が出てきたときにどう扱うか、確信が持てない情報をどう扱うか、といった「際どい場面」でのふるまいでした。

**governanceの強いrepoでは、Opus 4.7が一番安定していました。** ADR・ticket・truth sourceといった「守るべき前提」が多い環境だと、迎合せずに一貫して判断を切り分けられる4.7の特性がそのまま活きてきます。

**大局判断ではOpus 4.8が強いです。** 前提そのものを疑う、過去の設計判断を横断的に参照する、といった「一段上から見る」タスクでは4.8の洞察が際立っていました。ただ断定が強く出ることもあるので、鵜呑みにせず4.7で裏を取る、という組み合わせがちょうどよく機能しました。

**小粒な改善案出しなら、Opus 4.6でも十分実用的です。** governanceへの接続は弱いものの、コードレベルの目利きは悪くありません。「まず改善候補を洗い出したい」という初期フェーズでは、身軽さも含めて普通に選択肢になります。

**1モデルに全部任せるより、役割分担したほうが結果は安定します。** これが今回の一番実務的な結論だと思っています。「このタスクではどの失敗パターンを一番避けたいか」でモデルを選ぶ、という発想のほうが、「一番良いモデルに全部投げる」よりも結果的にうまくいきました。

## 注意点

この検証結果を読む上で、先に断っておきたい点がいくつかあります。

- サンプル数は限定的で、7種類のプロンプト・1つのrepoでの観察にすぎません
- 結果はこのrepo固有のもので、ADRやgovernanceの薄いrepoでは傾向が変わる可能性があります
- プロンプトの書き方や会話の文脈によって、同じモデルでも挙動が変わることがあります
- モデル名やバージョンごとの特性は今後のアップデートで変わる可能性が十分にあります

つまりこの記事は「Opus 4.7が常に最強」という主張ではなく、「今回のこのrepo・このプロンプト群では、こういう傾向が見えた」という限定付きの実験レポートとして読んでいただけるとうれしいです。

## まとめ

Claude Opus 4.6 / 4.7 / 4.8 を、governanceの強いShopifyアプリテンプレートリポジトリで比較してみたところ、次のような役割分担が見えてきました。

- **Opus 4.6**: 実装者型です。小粒なリファクタ候補出しに強い一方、repoのルールへの接続はやや弱めでした
- **Opus 4.7**: テックリード型です。ADR・ticket・truth sourceを踏まえた判断が安定しており、日常業務の本命でした
- **Opus 4.8**: アーキテクト型です。大局判断や前提矛盾の検出に強い一方、断定や脱線には注意が必要でした

「どれが一番賢いか」ではなく「どのタスクでどう失敗しやすいか」を軸に見ることで、モデル選びがだいぶ具体的になった気がしています。governanceの強いrepoで働いているエンジニアの方なら、同じような軸で自分のリポジトリを使って試してみる価値はあると思います。

## 今後の展望

今回はレビューや調査、設計相談といった「言葉で判断する」タスクを中心に比較しました。次にやるなら、各モデルに同じ実装修正タスクを投げて、実際のdiffとテスト結果で比較してみたいです。判断力や説明の質だけでなく、「実際に動くコードをどれだけ安全に書き換えられるか」まで見て初めて、使い分けの結論はもう一段確かなものになる気がしています。
