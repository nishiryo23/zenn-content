## はじめに

前提として私は、AIで早く仕事を終わらせて空いた時間に自己研鑽をする——わけでもなく、ただただサボりたいがために日々AIの活用法を模索しているフロントエンドエンジニアです。

受託開発のチームでFigmaのデザインカンプをマークアップする仕事をしていて、最近はClaude CodeやCodexからFigma MCPを使って実装する流れがほぼ定着しました。体感で9割くらいはデザインを再現できていて、本当にAIの凄さに驚いているところです。

ただ、最近ずっと頭を悩ませていることがあります。

Figma MCPでデザインの値だけをAIに渡して実装させると、**なんの機能も搭載されていない、ただただ見た目を再現しただけの状態**ができあがるんです。

「バラバラに散らばっているコンテキスト」をどう集約してAIに渡すか——最適解はまだ絶賛模索中ではあるのですが、その試行錯誤の過程のアウトプットにお付き合いいただければ幸いです。

## 結論

先に結論です。

**「AIにコンテキストを渡す」のではなく、「コンテキストのあるところにAIをぶち込む」。**

いちいちプロンプトに仕様や要件を書いて渡すのではなく、情報が自然に蓄積される場所にAIがアクセスできる仕組みを作る。発想としてはそれだけなのですが、これが思っている以上に難しい。

## 前提——チームのAI活用体制とマークアップの現在地

自分たちのチームでは、クライアントごとにGitHub上でAIエージェント用のリポジトリを管理しています。議事録、提案書、要件定義書——こういったドキュメントがリポジトリに蓄積されていき、作業時にはAIがそのまま参照できる。コンテキストを揃えた上でAIを打ち込むイメージです。

マークアップの構成はこうです。

- **ツール**: Claude Code / Codex
- **デザイン取得**: Figma MCP（MCPサーバー経由でFigmaのデザイン値を取得）

簡単なLPの静的マークアップならこれで十分です。問題は、Figma以外にもAIに渡したいコンテキストがあるときです。

たとえばShopifyやecforceなどのEC構築では、見た目の再現に加え、カートシステムのAPIを叩く動的な処理が入ります。具体例です。

- サイトTOPのあるセクションに、管理画面から毎日おすすめ商品を手動設定したい
- GA4のセッション数が多い商品詳細の順で商品を並べたい

こういった仕様はクライアントによって独自の構築になります。そしてこれらの情報は、ディレクターやデザイナーがFigmaのコメント機能やアノテーション機能を使ってテキストで残していることが多い。

ところが、**Figma MCPにはコメントを取得する機能がありません**（2026年7月時点）。アノテーションも実用上は十分に取得できる状態ではなく、Figma上のテキスト情報をAIに届ける手段がないのが現状です。

つまりFigmaのコンテキストだけでAIに実装させると、見た目は再現されているのに仕様は一切反映されていない状態ができあがる。修正指示もFigma上に書かれるので、それをいちいちAIに伝えるのも手間がかかります。

もちろんプロンプトに仕様を書けばいい話です。でもせっかくAIを使うなら、自分の性格上、最大限恩恵を受けたい。この「バラバラに散らばっているコンテキスト」をどう集約するかが、最近の自分の課題でした。

## アプローチ①　スプレッドシートに全情報を集約する

最初に考えたのは、スプレッドシートへの集約です。

プロジェクトには必ずと言っていいほどスプレッドシートがあって、スケジュールや要件がまとまっています。ここにFigma上の仕様テキストも持ってくれば、AIが一箇所で読めるのではないかと。デザインの値はFigmaに持たせるとして、動的処理の仕様や修正指示などのテキスト情報を一元管理する発想です。

ただ、すぐに壁にぶつかりました。

スプレッドシートでは画像が扱いにくく、「この仕様はFigma上のどのデザインカンプの、どのパーツやセクションのことを指しているのか」が分かりづらい。ディレクターやデザイナーの目線で見れば、Figma上に直接テキストを残す方が楽だし、視覚的にも人間にとっては分かりやすい。

つまり、**人間のワークフローを変えないと成立しない方法だった**。これは頓挫しました。

## アプローチ②　REST API × Figma MCPでskillを作る

次に試したのがFigma REST APIです。MCPでは取得できなかったコメント、アノテーション、キャンバス上の付箋——これらがREST APIでは取れます。

### MCPだけでは足りない理由

Figma MCPが取得できるのは、ノードのデザイン値（色、サイズ、フォント等）です。一方で以下は取得できません。

- **コメント**: 修正指示や仕様の議論
- **Dev Modeアノテーション**: 開発者向けの補足メモ
- **キャンバス上の付箋**: 仕様メモ、注意書き

デザインの「見た目」は取れるのに、デザインの「意図」が取れない。これが問題の正体です。

### skillの設計思想

REST APIでコメントは取得できるのですが、そのまま使うと「全ページの全コメント」がまとめて返ってきます。「この画面のこのパーツに紐づいているコメント」のようなフィルタリングはAPIだけでは解決できません。

そこで、Figma MCP × REST APIの組み合わせで、デザイン値とコメントをそれぞれ取得してからAIにマッピングさせるClaude Code用のskillを開発しました。

設計で意識したのは3つです。

**1. 人間のワークフローを変えない。**
Figmaは「書き込みの場」、リポジトリは「AIが読む場」。同期は常にFigma→リポジトリの一方向です。ディレクターやデザイナーがFigmaにコメントを書く既存の流れは一切触りません。アプローチ①で学んだ教訓がここに効いています。

**2. 取得は機械、分類だけAI。**
Node.jsのスクリプトがREST APIを叩いて全件取得・画面ごとの振り分け・位置計算まで機械的にやります。AIが担当するのは、取得済みのコメントを「修正指示 / 仕様 / 雑談」に分類する蒸留だけ。AIに任せる範囲を最小限に絞ることで、再現性を上げています。

**3. 黙って落とさない。**
どの画面にも紐付かなかったコメントは「unmatched」として件数を報告します。「全部取れました」と嘘をつかない。これは地味ですが、信頼して運用するためには必須でした。

### 仕組みの概要

流れはこうです。

1. 設定ファイルに対象画面のFigma URLを登録する
2. スクリプトがREST APIで4種のデータを取得する——コメント全文、Dev Modeアノテーション、キャンバス上の付箋テキスト、画面のスクリーンショット
3. 各コメントをノードツリーで画面に自動マッピングする（コメントが紐づいているノードIDが、どの画面のサブツリーに属するかで振り分け）
4. 画面ごとにMarkdownファイルとして出力する
5. AIが新着コメントを読み、修正指示と仕様に蒸留する

スプレッドシートのアプローチとの決定的な違いは、**人間がFigmaに書き込む行為がそのまま、AIが読めるコンテキストになる**ところです。人間に新しい作業を課さず、情報の流れだけを整えました。

### 結果

思っていたよりマッピングがしっかり紐づいていました。AIに参照させてマークアップ修正をかけたところ、8割くらいは意図した動きをしてくれた。

ただ、万能ではありません。残り2割はマッピングの段階で漏れている。シンプルな「背景色を#◯◯に変更」という指示すらスルーされることがある。skillの精度がボトルネックです。

## 反論と限界

「プロンプトに毎回仕様を書けばいいのでは？」——一番シンプルな反論はこれだと思います。

正直、それでも回ります。1画面の修正なら、Figmaのコメントを目で見てプロンプトに貼れば済む話です。

でもそれは、案件が変わるたびにゼロから手作業をやり直すことでもある。コンテキストの集約を仕組みとして持っておけば、次の案件でも初日からAIが文脈を持って動ける。

今はまだskillの精度は完璧ではないし、やりすぎに見えるかもしれません。でも、**AIを業務で使い倒すための限界みたいなところまで挑戦する意味があるフェーズ**だと思っています。今ここで仕組みを作っておくことが、半年後に確実に効いてくる。そう信じているところです。

現時点での限界も正直に書いておきます。

- マッピング精度は8割。残り2割の原因はskillの振り分けロジックにある
- Figma MCP自体がコメントやアノテーションの取得に対応すれば、このskillの大半は不要になる可能性がある
- 「全ページ×全コメント」を画面単位に紐づける処理は、コメント数が増えるほど精度が落ちるリスクがある

## おわりに

正直なところ、最適解はまだ見つかっていません。

ただ、2つのアプローチを試して確信したことがあります。**AI活用のボトルネックは、AIの能力ではなく、コンテキストの設計です。**

「コンテキストのあるところにAIをぶち込む」——この発想はマークアップに限った話ではないと思っています。議事録、仕様書、修正指示、どの業務でも「情報はあるのにAIに届いていない」場面は山ほどある。

同じように「AIに何をどう渡すか」で頭を悩ませている方がいたら、ぜひコメントで聞かせてください。
